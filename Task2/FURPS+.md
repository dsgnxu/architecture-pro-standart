# FURPS+ таблица требований


| Код | Требования | Комментарий |
|-----|------------|-------------|
| **F** | **Функциональные (Functionality)** | |
| F1 | На сайте потенциальный клиент видит список доступных депозитов с актуальными ставками | **MVP.** Сайт (PHP, React.js) сейчас только маркетинговый — нужна интеграция с источником ставок |
| F2 | На сайте клиент подаёт заявку на депозит| **MVP.** После отправки с клиентом связывается менеджер кол-центра. Заявка передаётся в систему КЦ (React, Spring Boot, PostgreSQL) |
| F3 | Менеджер кол-центра видит заявку и может предложить особые условия по депозиту | **MVP.** Система КЦ уже интегрирована с АБС |
| F4 | В интернет-банке клиент видит депозиты и накопительные счета с актуальными и персонализированными ставками | **MVP → 6 мес.** Предложения банка - выгодные ставки; персонализация для клиента |
| F5 | В интернет-банке клиент подаёт заявку на открытие депозита или накопительного счёта, указав счёт и сумму | **MVP** — заявка; **6 мес.** — открытие счёта |
| F6 | Операция в интернет-банке подтверждается СМС-кодом | **MVP.** СМС в ядре интернет-банка (подрядчик) — реализовать силами команды банка, без доработки ядра |
| F7 | Ставки по депозитам ведутся автоматизированно (не в Excel); доступны бэк-офису депозитов и кредитов | **MVP.** Сейчас только XLS и почта. Целесообразно реализовать в АБС (Delphi, Oracle) |
| F8 | Процесс согласования ставки по заявке автоматизирован в АБС | **MVP → 12 мес.** Заменяет ручной процесс (почта + Excel). На MVP бэк-офис ещё участвует |
| F9 | На этапе MVP заявку на депозит обрабатывает менеджер бэк-офиса, подтверждая условия в АБС | **MVP.** Полная автоматизация — цель на 12 месяцев (+R11) |
| F10 | Клиент получает СМС после подтверждения ставки и открытия депозита/накопительного счёта | **MVP.** Через существующий СМС-шлюз |
| F11 | Клиент полностью автоматически открывает депозит или накопительный счёт в интернет-банке без участия бэк-офиса | **12 мес.** Финальное состояние; снижение операционных издержек |
| F12 | При обращении в отделение депозит или накопительный счёт открывается силами только фронт-офиса, без бэк-офиса | **12 мес.** Ставки и условия уже согласованы в системе; бэк-офис исключён из процесса |
| F13 | На сайте клиент видит предложения по кредитам; полные условия хранятся в Кредитном конвейере | **MVP.** Кредитный конвейер (Camunda BPM, Oracle, Java) — источник полных условий |
| F14 | На сайте клиент подаёт заявку на кредит | **MVP.** О клиенте известно только то, что есть в БКИ |
| F15 | Для нового клиента с сайта предварительный скоринг выполняется по данным БКИ (REST API) | **MVP.** Внутренних данных в АБС нет; скоринг (Python, Flask, PostgreSQL) — онлайн |
| F16 | Клиент получает предварительное решение об одобрении/отказе в течение одного дня | **MVP → 12 мес.** Цель бизнеса: решение за день, а не за неделю |
| F17 | После заявки на кредит с сайта клиент получает СМС о визите в отделение для идентификации и получения кредита | **MVP.** |
| F18 | Менеджер отделения продолжает ту же заявку на кредит, если клиент получил предодобрение на сайте | **MVP.** Единый идентификатор заявки и сквозной статус между каналами |
| F19 | В интернет-банке клиент видит предодобренные кредитные предложения и сразу понимает, может ли потенциально получить кредит | **MVP.** Механика конкурентов: предодобрение → быстрая заявка |
| F20 | Клиент заполняет быструю заявку на кредит по предодобренному предложению в интернет-банке | **MVP.** Такая заявка исполняется в течение одного дня, а не недели |
| F21 | При подаче заявки в интернет-банке выполняется повторный скоринг; при неизменившейся ситуации — упрощённая процедура | **MVP.** Ускоренная обработка сотрудниками |
| F22 | Заявка на кредит в интернет-банке подтверждается СМС-кодом | **MVP.** Без доработки ядра подрядчика |
| F23 | Клиент получает СМС об изменении статуса заявки на кредит | **MVP.** |
| **U** | **Удобство использования (Usability)** | |
| U1 | Интерфейс сайта и интернет-банка максимально удобен для клиента | Бизнес-ожидание на всех этапах |
| U2 | Клиент видит актуальные ставки и условия без устаревших данных | Критично для конкурентного преимущества по ставкам |
| U3 | Предодобренное кредитное предложение и быстрая заявка доступны в 1–2 шага без лишних полей | **MVP.** Снижает отказ от заявки; аналог конкурентов |
| **R** | **Надёжность (Reliability)** | |
| R1 | Все сервисы работают 24/7 с доступностью 99,9% | Бизнес-требование; команда интернет-банка считает его пока невыполнимым |
| R2 | При сбое основного ЦОД сервисы интернет-банка доступны через резервный ЦОД | Сейчас один ЦОД; резервный есть, но не задействован |
| R3 | Избежать прямой онлайн-работы интернет-банка с API/БД АБС в новых процессах | БД АБС перегружена; массовые онлайн-заявки угрожают работоспособности |
| R4 | Взаимодействие интернет-банка и АБС в новых процессах — асинхронное (очередь сообщений) | При сбое одной системы вторая продолжает принимать заявки |
| R5 | Система кредитного скоринга не получает дополнительную нагрузку от предрасчётов | Скоринг критичен и перегружен в рабочие часы |
| R6 | Заявки по предодобренным предложениям обрабатываются в течение одного рабочего дня | **MVP → 12 мес.** Конфликт с AS-IS: batch АБС ↔ Конвейер 1 раз/сут; нужен обходной маршрут (+R9, +R10) |
| **P** | **Производительность (Performance)** | |
| P1 | Отклик по всем операциям в интернет-банке — миллисекунды | Бизнес-ожидание |
| P2 | Загрузка справочных данных — не более 1 секунды | Текущая проблема при платежах |
| P3 | Горизонтальное масштабирование и распределение запросов между серверами и ЦОД | Конвейер и скоринг уже масштабируются горизонтально |
| P4 | Кэширование справочных данных (ставки, продукты) в каналах самообслуживания | Снижает нагрузку на АБС; важно при росте онлайн-продуктов |
| **S** | **Поддерживаемость (Supportability)** | |
| S1 | Документация для дальнейшего расширения: MVP → 6 мес. → 12 мес. | Заложить эволюцию от заявки к полной автоматизации |
| S2 | Новый функционал интернет-банка (депозиты, накопительные счета, кредиты) — как микросервисы | Поэтапный перевод без переписывания монолита ASP.NET MVC |
| S3 | СМС-подтверждение и уведомления — силами команды банка, без ядра подрядчика | Снижение зависимости от подрядчика |
| S4 | Централизованное ведение ставок и условий продуктов с версионированием | Исключает рассогласование между сайтом, ИБ и АБС на всех этапах |
| S5 | Архитектура MVP проектируется с учётом перехода к полной автоматизации открытия счетов через 12 месяцев | Иначе потребуется переработка интеграций и процессов |
| **+R** | **+ Ограничения (Restrictions)** | |
| +R1 | Использовать технологии банка: MS SQL, Oracle, .NET, Delphi, Java Spring Boot, React.js, PHP | Соответствует текущему стеку |
| +R2 | Новые технологии совместимы с существующими платформами разработки | |
| +R3 | Внутри банка есть экспертиза по используемым технологиям | |
| +R4 | Для очередей сообщений — Kafka (на перспективу) | Текущая платформа интернет-банка несовместима с Kafka |
| +R5 | АБС масштабируется только вертикально (Oracle) | Горизонтальное масштабирование АБС невозможно |
| +R6 | Интернет-банк — монолит ASP.NET MVC; ядро у подрядчика | Новый функционал выносить за пределы ядра |
| +R7 | Шифрование трафика (TLS/HTTPS) при передаче данных с сайта и из интернет-банка | ФИО, телефон, паспортные данные |
| +R8 | Бэк-офис депозитов и кредитов не имеет доступа к данным друг друга | AS-IS: изоляция через письма; при централизации ставок — разграничение |
| +R9 | Обмен АБС ↔ Кредитный конвейер — пакетно 1 раз/сут; учащение невозможно | Для онлайн-каналов и SLA «решение за день» нужен прямой маршрут |
| +R10 | Онлайн-скоринг с сайта и интернет-банка — REST API к скорингу и БКИ | Batch через АБС не подходит для мгновенных решений |
| +R11 | **MVP:** только онлайн-заявка на депозит, бэк-офис обрабатывает по старому процессу | Граница scope MVP |
| +R12 | **6 мес.:** открытие депозита/накопительного счёта в ИБ; бэк-офис может ещё участвовать | Промежуточное состояние |
| +R13 | **12 мес.:** полная автоматизация открытия в ИБ и в отделении без бэк-офиса | Финальное состояние; цель по сокращению издержек |
| +R14 | На сайте о клиенте-кредитополучателе известны только данные БКИ | Ограничивает глубину скоринга для новых клиентов с сайта |
