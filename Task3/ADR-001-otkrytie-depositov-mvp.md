### **Решение**

Приведите диаграммы контекста и контейнеров в модели C4. Опишите там основные компоненты и интеграции всех элементов решения.

Также опишите, какой логикой вы руководствовались в ходе принятия решений и выбора технологий. Не забывайте, что необходимо учесть все функциональные и нефункциональные требования.

**Выбранный подход:** асинхронная event-driven интеграция через **Kafka** с микросервисами команды банка **вне ядра монолита интернет-банка** (вариант 3).

**Логика принятия решений:**
- Прямая работа с Oracle АБС из ИБ недопустима (R3, перегрузка БД).
- Kafka обеспечивает асинхронность и отказоустойчивость (R4); ABS Integration Adapter — consumer.
- Deposit Service (.NET) и SMS Confirmation Service — вне ядра подрядчика (S2, S3, +R6).
- Ставки — единый источник в АБС, REST API + кэш в каналах (S4, P4, U2).
- Сайт и КЦ используют существующий стек (+R1); MVP не меняет процесс идентификации в отделении (+R11).

#### Диаграмма контекста (C4, Level 1)

Файл: [`c4-context-deposit-mvp.drawio`](c4-context-deposit-mvp.drawio)

Система **«Цифровое открытие депозитов (MVP)»** — онлайн-каталог, приём заявок, передача в АБС.

**Акторы:** потенциальный клиент, клиент банка, оператор КЦ, менеджер бэк-офиса депозитов.

**Системы:** Сайт (PHP, React.js), Интернет-банк (ASP.NET MVC), Система КЦ (Spring Boot, React), АБС (Delphi, Oracle), СМС-шлюз.

**Ключевые интеграции:**
- Сайт → АБС: REST, каталог ставок (кэш).
- Сайт → КЦ: REST, заявка.
- ИБ → Kafka → АБС: асинхронная заявка.
- КЦ → АБС: существующая интеграция.
- АБС → СМС-шлюз: уведомления.

#### Диаграмма контейнеров (C4, Level 2)

Файл: [`c4-container-deposit-mvp.drawio`](c4-container-deposit-mvp.drawio)

| Система | Контейнер | Назначение |
|---------|-----------|------------|
| **Сайт** | Веб-приложение сайта | Каталог депозитов, форма заявки |

- **Ставки:** бэк-офис вводит в АБС → REST API → кэши каналов.

### **Альтернативы**

Опишите здесь наиболее важные альтернативные решения.

1. **Прямая интеграция каналов с БД АБС** — как сейчас у интернет-банка для платежей. Отклонено: нарушает R3, перегружает Oracle.
2. **Синхронный REST API поверх АБС** — все вызовы online. Отклонено: при сбое АБС каналы блокируются; не решает R4.
3. **Асинхронная интеграция через Kafka + микросервисы вне монолита ИБ** — **принято.** Баланс R3, R4, S2, S3.

### **Недостатки, ограничения, риски**

Подробно опишите здесь недостатки, ограничения и риски выбранного решения.

**Недостатки:**
- Вводится Kafka — новый компонент для эксплуатации и мониторинга.
- Eventual consistency: клиент видит статус заявки с задержкой до обработки в АБС.
- На MVP бэк-офис участвует в каждой заявке — операционные издержки сохраняются (+R11).
- Координация двух команд ИБ (банк и подрядчик) по точкам интеграции с монолитом.

**Ограничения:**
- Согласование особых ставок (F8) на MVP частично ручное — автоматизация на 12 месяцев.
- Полностью автоматическое открытие депозита (F11, +R13) — вне scope MVP.
- Платформа ИБ несовместима с Kafka напрямую — нужен адаптер (+R4).

**Риски:**
- Задержка согласования точек расширения монолита с подрядчиком.
- Рост очереди Kafka при длительном недоступности АБС.
- Рассогласование кэша ставок при сбое TTL-обновления (митigation: версионирование S4).
