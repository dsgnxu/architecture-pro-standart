# ADR-001: Концептуальная архитектура открытия депозитов (MVP)

## Статус

Принято

## Дата

2026-06-06

## Контекст

Банк «Стандарт» цифровизирует депозитное направление. На этапе **MVP** (+R11) клиент может подать **онлайн-заявку** на депозит через сайт или интернет-банк; фактическое открытие счёта и согласование особых условий по-прежнему выполняет **бэк-офис депозитов** в АБС по существующему процессу (F8, F9, +R11). Полная автоматизация без участия бэк-офиса — цель на 12 месяцев (F11, +R13).

**Текущее состояние (AS-IS):**
- Депозиты открываются только в отделении; сайт — маркетинговый, интернет-банк не участвует в депозитах.
- Ставки рассчитываются в Excel и передаются по почте между бэк-офисами депозитов и кредитов (+R8).
- Интернет-банк интегрирован с БД АБС напрямую; БД АБС перегружена (R3).
- Интернет-банк — монолит ASP.NET MVC, ядро у подрядчика (+R6); новый функционал нельзя встраивать в ядро (S2, S3).
- Система кол-центра (React, Spring Boot, PostgreSQL) уже интегрирована с АБС.

**Силы, влияющие на решение:**
- Бизнес хочет онлайн-заявки на сайте и в ИБ уже на MVP (F1–F6, F2, F5).
- Команда ИБ требует асинхронного взаимодействия с АБС и отказа от прямых запросов к БД Oracle (R3, R4).
- Kafka на перспективу (+R4), но текущая платформа ИБ с ней несовместима — нужен совместимый брокер сообщений.
- Ставки должны вестись централизованно в АБС, а не в Excel (F7, S4).
- СМС-подтверждение и уведомления — силами команды банка, без доработки ядра подрядчика (F6, F10, S3).
- Архитектура MVP должна допускать эволюцию к полной автоматизации через 6 и 12 месяцев (S1, S5, +R12, +R13).

## Use Cases (MVP — открытие депозитов)

| ID | Use Case | Актор | Связанные требования | Канал / система |
|----|----------|-------|----------------------|-----------------|
| UC-1 | Просмотр каталога депозитов с актуальными ставками | Потенциальный / действующий клиент | F1, F4, U2, P4 | Сайт, ИБ |
| UC-2 | Подача заявки на депозит с сайта (ФИО, телефон) | Новый клиент | F2, +R7 | Сайт → Система КЦ |
| UC-3 | Обработка заявки с сайта, предложение особых условий | Оператор кол-центра | F3 | Система КЦ → АБС |
| UC-4 | Просмотр депозитов и накопительных счетов со ставками | Клиент банка | F4, U2 | ИБ |
| UC-5 | Подача заявки на открытие депозита (счёт, сумма) | Клиент банка | F5, +R11 | ИБ |
| UC-6 | Подтверждение заявки СМС-кодом | Клиент банка | F6, S3 | ИБ (сервис банка) |
| UC-7 | Автоматизированное ведение ставок по депозитам | Менеджер бэк-офиса депозитов | F7, S4, +R8 | АБС |
| UC-8 | Обработка онлайн-заявки, подтверждение условий | Менеджер бэк-офиса депозитов | F8*, F9, +R11 | АБС |
| UC-9 | Уведомление клиента о подтверждении ставки / открытии | Клиент | F10 | АБС → СМС-шлюз |

\* На MVP согласование особых ставок остаётся ручным (почта между подразделениями); автоматизация F8 — на 12 месяцев.

**Вне scope MVP:** UC «Полностью автоматическое открытие депозита в ИБ без бэк-офиса» (F11, +R13); UC «Открытие депозита силами только фронт-офиса» (F12).

## Рассмотренные варианты

1. **Прямая интеграция каналов с БД АБС** — как сейчас у интернет-банка для платежей.
2. **Синхронный REST API поверх АБС** — новый API-слой для онлайн-заявок.
3. **Асинхронная интеграция через очередь сообщений + микросервисы вне монолита ИБ** — новые сервисы банка, брокер сообщений между каналами и АБС.

## Решение

Мы реализуем **вариант 3**: асинхронную event-driven интеграцию через **Kafka** (+R4) с выносом депозитного функционала в **отдельные микросервисы** команды банка вне ядра монолита интернет-банка. Для совместимости с текущей платформой ИБ на MVP допускается промежуточный адаптер между монолитом и Kafka.

### Участвующие IT-команды

| Команда | Ответственность в MVP |
|---------|----------------------|
| **Команда цифровой трансформации** (10 чел.) | Координация, архитектура, интеграционный контракт, сайт (депозитный каталог и форма заявки) |
| **Команда интернет-банка (банк)** (10 чел.) | Микросервис заявок на депозит, сервис СМС-подтверждения, кэш ставок, интеграция с монолитом ИБ |
| **Команда интернет-банка (подрядчик)** (10 чел.) | Точки расширения монолита (виджет/iframe/API gateway), без изменений ядра |
| **Команда АБС** (20 чел.) | Модуль ставок депозитов, модуль обработки заявок, адаптер очереди сообщений, REST API каталога ставок |
| **Команда системы кол-центра** | Приём заявок с сайта (существующая интеграция с АБС) |
| **Команда сайта / маркетинга** | UI каталога и формы заявки (PHP, React.js) совместно с цифровой трансформацией |

### Диаграмма контекста (C4, Level 1)

Файл: [`c4-context-deposit-mvp.drawio`](c4-context-deposit-mvp.drawio)

Система **«Цифровое открытие депозитов (MVP)»** — программная система банка, обеспечивающая онлайн-каталог депозитов, приём заявок и передачу их на обработку в АБС.

**Акторы:**
- **Потенциальный клиент** — просматривает депозиты и подаёт заявку на сайте (UC-1, UC-2).
- **Клиент банка** — просматривает депозиты и подаёт заявку в интернет-банке (UC-1, UC-4, UC-5, UC-6).
- **Оператор кол-центра** — обрабатывает заявки с сайта (UC-3).
- **Менеджер бэк-офиса депозитов** — ведёт ставки и обрабатывает заявки в АБС (UC-7, UC-8).

**Внешние системы:**
- **Сайт банка** — канал привлечения (PHP, React.js).
- **Интернет-банк** — канал самообслуживания (ASP.NET MVC, MS SQL).
- **Система кол-центра** — CRM заявок (React, Spring Boot, PostgreSQL).
- **АБС** — учётная система, ставки, обработка заявок (Delphi, Oracle).
- **СМС-шлюз** — доставка СМС клиентам.

**Ключевые связи:**
- Клиенты → Сайт / ИБ: HTTPS (TLS, +R7).
- Сайт → АБС: REST, каталог ставок (read-only, с кэшем на стороне сайта).
- Сайт → Система КЦ: REST, заявка на депозит (UC-2).
- ИБ → Kafka → АБС: асинхронная передача заявок (UC-5, UC-6).
- Система КЦ → АБС: существующая интеграция (UC-3).
- АБС → СМС-шлюз: уведомления (UC-9).

### Диаграмма контейнеров (C4, Level 2)

#### Сайт банка [PHP, React.js]

| Контейнер | Назначение | Интеграции |
|-----------|------------|------------|
| **Веб-приложение сайта** | Маркетинговые страницы, каталог депозитов, форма заявки | → API каталога ставок (АБС); → API системы КЦ |

#### Интернет-банк [ASP.NET MVC, MS SQL] — детализация

| Контейнер | Назначение | Интеграции |
|-----------|------------|------------|
| **Монолит ИБ (ядро подрядчика)** | Аутентификация, счета, платежи; точка встраивания UI депозитов | → Deposit Service (REST); read-only к MS SQL |
| **Deposit Service** [.NET, микросервис банка] | Каталог депозитов для клиента, приём и валидация заявки, оркестрация | → Rates Cache; → SMS Confirmation Service; → Kafka (publish); ← API каталога ставок (АБС) |
| **SMS Confirmation Service** [.NET, микросервис банка] | Генерация и проверка СМС-кодов для заявок (UC-6) | → СМС-шлюз; ← Deposit Service |
| **Rates Cache** [Redis / in-memory] | Кэш справочника ставок и продуктов (P4) | ← ABS Rates API (TTL-обновление) |
| **БД интернет-банка** [MS SQL] | Данные монолита, статусы заявок (локальная проекция) | ← Deposit Service |

#### АБС [Delphi, Oracle] — детализация

| Контейнер | Назначение | Интеграции |
|-----------|------------|------------|
| **Клиентское приложение АБС (Delphi)** | Рабочие места фронт- и бэк-офиса | → Oracle DB |
| **Модуль ставок депозитов** [Delphi, новый] | Ведение ставок и условий продуктов, версионирование (F7, S4); замена Excel | → Oracle DB; разграничение доступа депозиты/кредиты (+R8) |
| **Модуль обработки заявок** [Delphi, новый] | Очередь онлайн-заявок для бэк-офиса, подтверждение условий (UC-8, F9) | → Oracle DB; → СМС-шлюз (UC-9) |
| **ABS Integration Adapter** [Java/Spring Boot или Delphi-сервис] | REST API каталога ставок; consumer Kafka | → Kafka; → Oracle DB; ← Deposit Service, CC System |
| **БД АБС** [Oracle] | Клиенты, счета, депозиты, ставки, заявки | ← все модули АБС |

### Основные потоки (MVP)

**Поток A — заявка с сайта (UC-1 → UC-2 → UC-3 → UC-8 → UC-9):**
1. Клиент видит ставки (REST к ABS Rates API, кэш на сайте).
2. Клиент отправляет заявку → Система КЦ.
3. Оператор КЦ обрабатывает заявку → АБС (существующая интеграция).
4. Бэк-офис подтверждает условия в АБС → СМС клиенту.
5. Клиент приходит в отделение для идентификации и открытия.

**Поток B — заявка из интернет-банка (UC-4 → UC-5 → UC-6 → UC-8 → UC-9):**
1. Клиент видит депозиты (Deposit Service → Rates Cache / ABS Rates API).
2. Клиент заполняет заявку → Deposit Service.
3. SMS Confirmation Service отправляет и проверяет код (UC-6).
4. Deposit Service публикует событие `DepositApplicationSubmitted` в Kafka.
5. ABS Integration Adapter сохраняет заявку; бэк-офис обрабатывает в модуле заявок.
6. После подтверждения АБС отправляет СМС через шлюз.

**Поток C — ведение ставок (UC-7):**
1. Менеджер бэк-офиса вводит/обновляет ставки в модуле ставок АБС.
2. ABS Rates API отдаёт актуальные данные каналам; кэши обновляются по TTL.

## Последствия

### Положительные

- Снижение нагрузки на БД АБС: каналы не обращаются к Oracle напрямую (R3).
- При сбое АБС ИБ продолжает принимать заявки в очередь (R4).
- Централизованные ставки исключают рассогласование между сайтом, ИБ и АБС (S4, U2).
- Новый функционал вне ядра подрядчика — меньше зависимости и проще эволюция к F11 (+R13).
- Архитектура готова к добавлению автоматического открытия счёта (6 мес., +R12) без переработки интеграций (S5).
- Кэширование ставок обеспечивает отклик в миллисекундах для справочных данных (P1, P2, P4).

### Отрицательные

- Вводится новый компонент — Kafka; потребуется эксплуатация и мониторинг. На MVP может потребоваться адаптер совместимости с монолитом ИБ (+R4).
- Eventual consistency: клиент видит статус заявки с задержкой до обработки сообщения АБС.
- На MVP бэк-офис всё ещё участвует в каждой заявке — операционные издержки сохраняются (+R11).
- Согласование особых ставок (F8) остаётся частично ручным до 12 месяцев.
- Две команды ИБ (банк и подрядчик) — нужна координация точек интеграции с монолитом.

### Нейтральные

- Система КЦ и сайт дорабатываются в рамках существующего стека (+R1).
- Процесс открытия в отделении для новых клиентов с сайта не меняется (идентификация обязательна).

## Подтверждение

- Диаграммы C4 покрывают все Use Cases MVP (UC-1 … UC-9).
- Архитектурное ревью с командами ИБ, АБС и цифровой трансформации.
- Проверка NFR: отсутствие прямых запросов ИБ к БД АБС в новых процессах; TLS на внешних каналах.
