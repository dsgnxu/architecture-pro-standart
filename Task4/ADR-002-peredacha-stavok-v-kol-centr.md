### <a name="_u8xz25hbrgql"></a>**Нефункциональные требования**

| **№** | **Требование** |
| :-: | :- |
| 1 | **Доступность.** Справочник ставок в кол-центре банка доступен оператору круглосуточно. Если АБС на короткое время недоступна — показываются последние сохранённые ставки с пометкой о дате обновления. |
| 2 | **Время отклика.** Справочник открывается в интерфейсе КЦ не дольше 1 секунды — данные хранятся локально в кэше, без ожидания ответа от АБС при каждом звонке. |
| 3 | **Удобство и дизайн.** Оператор видит ставки в 1–2 клика прямо из карточки обращения, может быстро найти продукт по названию или сроку. Рядом с данными отображается дата актуальности, чтобы оператор не озвучил клиенту устаревшие условия. |
| 4 | **Безопасность и согласованность.** Для партнёрского КЦ файл передаётся только через защищённый SFTP; персональных данных клиентов в файле нет. Ставки на сайте, в КЦ банка и у партнёра совпадают — все берут данные из одного источника в АБС. |

### **Решение**

Приведите диаграммы контекста и контейнеров в модели C4. Опишите там основные компоненты и интеграции всех элементов решения.

Также опишите, какой логикой вы руководствовались в ходе принятия решений и выбора технологий. Не забывайте, что необходимо учесть все функциональные и нефункциональные требования.

**Выбранный подход:** расширение архитектуры Task 3 (вариант 3) — единый источник ставок в АБС, два канала доставки: **REST + кэш** для внутреннего КЦ и **SFTP + CSV** для партнёрского КЦ.

**Логика принятия решений:**
- ABS Rates API уже спроектирован в Task 3 — переиспользуем для внутреннего КЦ (минимальное расширение).
- Партнёр не поддерживает API (+R15) — единственный канал SFTP.
- Кэш в КЦ защищает Oracle от частых опросов (P5, +R5, R3).
- Rates Export Service подписывается на событие изменения ставок — автоматизация вместо Excel/почты (S4, F26).
- Файл содержит только публичные ставки без ПДн (F28, +R8).

Файл: [`c4-context-rates-cc.drawio`](c4-context-rates-cc.drawio)

### **Альтернативы**

Опишите здесь наиболее важные альтернативные решения.

1. **Ручная выгрузка Excel и отправка партнёру по почте** — продолжение AS-IS. Отклонено: не масштабируется, риск ошибок, нарушает S4.
2. **REST API ставок для партнёрского КЦ** — отклонено: партнёр не поддерживает API (+R15).
3. **REST для внутреннего КЦ + SFTP для партнёра** — **принято.** Единый источник, два канала.
4. **Отдельная «витрина ставок» (новая БД)** — отклонено: лишний компонент, риск рассогласования с АБС.

### **Недостатки, ограничения, риски**

Подробно опишите здесь недостатки, ограничения и риски выбранного решения.

**Недостатки:**
- Eventual consistency: партнёр может отставать до 24 ч (расписание импорта).
- Дополнительный компонент Rates Export Service и эксплуатация SFTP.
- Необходимо согласование формата CSV с внешним контрагентом.

**Ограничения:**
- Партнёрский КЦ — только файловый обмен (+R15).
- Персонализированные ставки в КЦ — этап 6 месяцев (F4).
- Внутренний КЦ не обращается к Oracle напрямую — только через REST + кэш.

**Риски:**
- Сбой SFTP: партнёр работает на последнем файле — нужен регламент (UC-15).
- Задержка согласования формата CSV с партнёром.
- Пиковая нагрузка на ABS Rates API при одновременном обновлении кэшей — митigation: TTL и P5.
