# ADR-002: Передача актуальных депозитных ставок в кол-центры (MVP)

## Статус

Принято

## Дата

2026-06-06

## Контекст

После запуска маркетинговой кампании по онлайн-депозитам ожидается рост обращений в кол-центр. Большинство клиентов банка «Стандарт» — пожилые люди; им сложнее ориентироваться в новых цифровых процессах, поэтому они будут уточнять условия открытия депозита по телефону.

**Проблема AS-IS:**
- Операторы кол-центра не имеют в своей системе актуального справочника ставок; при звонке до визита в отделение ставку озвучивают «из головы» или запрашивают у бэк-офиса.
- Ставки рассчитываются в Excel и передаются по почте между подразделениями (+R8 из Task2); в системе КЦ справочника нет.
- Для разгрузки внутреннего кол-центра (200 операторов) подключается **партнёрский кол-центр** (100 операторов) — внешняя информационная система без интеграции с АБС.
- Партнёр готов принимать ставки **только в виде файлов по SFTP**; REST/API-вызовы недоступны.

**Связь с ADR-001:**
- В MVP централизованное ведение ставок реализуется в **модуле ставок депозитов АБС** (UC-7, F7, S4); **ABS Rates API** уже отдаёт ставки сайту и интернет-банку через кэш (P4).
- Новый кейс расширяет MVP: те же актуальные ставки должны быть доступны **системе кол-центра банка** (онлайн) и **системе партнёрского кол-центра** (файловый обмен).

**Силы, влияющие на решение:**
- Единый источник истины по ставкам — АБС; дублирование в Excel недопустимо (S4, U2).
- Внутренний КЦ уже интегрирован с АБС (заявки); добавление read-only доступа к ставкам — минимальное расширение (+R1).
- Партнёрский КЦ — внешняя система; единственный согласованный канал — **SFTP + файлы** (ограничение контрагента).
- При изменении ставок операторы обоих КЦ должны видеть согласованные данные с сайтом и ИБ (U2).
- Нагрузка на Oracle АБС не должна расти из-за частых опросов КЦ (P4, +R5); КЦ не обращается к Oracle напрямую — по тому же принципу, что R3 для ИБ.

| Категория | Код | Требование |
|-----------|-----|------------|
| Usability | U4 | Справочник ставок в системе КЦ доступен в 1–2 клика из карточки обращения; поиск по названию продукта и сроку |
| Usability | U2 | Данные в КЦ не старше данных на сайте более чем на интервал синхронизации (≤ 15 мин для REST, ≤ 24 ч для SFTP-файла) |
| Reliability | R1 | Доступность справочника ставок для внутреннего КЦ — 99,9%; при недоступности АБС — отображение последнего кэша с пометкой |

| Performance | P2 | Загрузка справочника в UI КЦ — не более 1 секунды (локальный кэш) |
| Performance | P5 | Частота опроса ABS Rates API системой КЦ — не чаще 1 раза в 15 мин (снижение нагрузки на АБС, +R5) |
| Supportability | S4 | Версия файла и REST-ответа совпадает с версией набора ставок в модуле АБС |
| Supportability | S6 | Формат файла ставок документирован и версионируется (v1: CSV UTF-8) |
| Restrictions | +R15 | Партнёрский КЦ не поддерживает API; только приём файлов по SFTP |
| Restrictions | +R1 | Стек: Java Spring Boot для сервиса выгрузки; SFTP — стандартный сервер банка или managed-сервис |
| Restrictions | +R8 | В выгрузку не попадают данные кредитного бэк-офиса; только депозитные продукты |

## Рассмотренные варианты

1. **Ручная выгрузка Excel из АБС и отправка партнёру по почте** — продолжение AS-IS.
2. **REST API ставок для партнёрского КЦ** — отклонён: партнёр не поддерживает API (+R15).
3. **Общий REST API для внутреннего КЦ + периодическая генерация файла и push на SFTP для партнёра** — единый источник в АБС, два канала доставки.
4. **Отдельная «витрина ставок» (новая БД), которую опрашивают все потребители** — лишний компонент и риск рассогласования.

## Решение

Принимаем **вариант 3**: расширяем архитектуру ADR-001 без нового источника данных.

### Внутренний кол-центр

- **Backend КЦ** получает ставки через существующий **ABS Rates API** (read-only), с **локальным кэшем** в PostgreSQL или Redis (TTL 15 мин, P5).
- **Frontend КЦ** — экран «Справочник депозитов» и виджет в карточке обращения (UC-10, UC-11, U4).

### Партнёрский кол-центр

- Новый контейнер **Rates Export Service** [Spring Boot] в контуре интеграции АБС:
  - подписывается на событие `DepositRatesUpdated` (из модуля ставок через ABS Integration Adapter / Kafka);
  - формирует файл **CSV UTF-8** (версия, дата, продукт, срок, мин. сумма, ставка %, валюта);
  - загружает на **SFTP-сервер банка** (исходящий каталог) или на endpoint партнёра;
  - пишет журнал выгрузок (UC-14, UC-15).
- Партнёр забирает файл по расписанию или получает push (UC-12, UC-13).

### Диаграмма контекста (C4, Level 1)

Файл: [`c4-context-rates-cc.drawio`](c4-context-rates-cc.drawio)

Система **«Распространение депозитных ставок в кол-центры»** дополняет MVP «Цифровое открытие депозитов».

**Акторы:**
- **Оператор кол-центра банка** — консультирует по ставкам (UC-10, UC-11).
- **Оператор партнёрского КЦ** — консультирует по ставкам из загруженного файла (UC-13).
- **Менеджер бэк-офиса депозитов** — изменяет ставки в АБС (UC-7 из ADR-001).
- **Администратор ИТ** — мониторинг выгрузок (UC-15).

**Системы:**
- **Система кол-центра банка** — REST к ABS Rates API.
- **АБС** — источник ставок.
- **Система партнёрского кол-центра** [External] — потребитель SFTP-файлов.
- **SFTP-сервер банка** [External Infrastructure] — точка обмена файлами.

### Диаграмма контейнеров (C4, Level 2)

Файл: [`c4-container-rates-cc.drawio`](c4-container-rates-cc.drawio)

| Система | Контейнер | Назначение | Интеграции |
|---------|-----------|------------|------------|
| **АБС** | Модуль ставок депозитов | Источник данных, событие об изменении | → Oracle DB; → Rates Export Service |
| **АБС** | ABS Integration Adapter | REST API каталога ставок (существующий) | ← Backend КЦ; ← Rates Export Service |
| **АБС** | **Rates Export Service** [Spring Boot, новый] | Генерация CSV, SFTP-upload, журнал | ← события ставок; → SFTP; → PostgreSQL журнал |
| **Система КЦ** | Backend КЦ | Кэш и API справочника для операторов | → ABS Rates API; → PostgreSQL (кэш) |
| **Система КЦ** | Frontend КЦ | UI справочника ставок | ← Backend КЦ |
| **Партнёрский КЦ** | CRM партнёра [External] | Импорт файла, работа операторов | ← SFTP (файл ставок) |

### Основные потоки

**Поток D — консультация во внутреннем КЦ (UC-10, UC-11):**
1. Оператор открывает карточку обращения → Frontend запрашивает Backend КЦ.
2. Backend отдаёт данные из локального кэша или обновляет из ABS Rates API (TTL 15 мин).
3. Оператор озвучивает клиенту актуальные условия.

**Поток E — доставка ставок партнёру (UC-14, UC-12, UC-13):**
1. Менеджер обновляет ставки в модуле АБС → публикуется `DepositRatesUpdated`.
2. Rates Export Service формирует `deposit_rates_v{version}_{date}.csv`.
3. Сервис загружает файл на SFTP; партнёр импортирует в CRM.
4. Оператор партнёра консультирует клиента по загруженным данным.

**Поток F — fallback по расписанию:**
1. Cron (ежедневно 06:00) генерирует файл даже без изменений — гарантия актуальности для партнёра (F26).

## Последствия

### Положительные

- Операторы обоих кол-центров консультируют по **тем же ставкам**, что сайт и ИБ (S4, U2).
- Внутренний КЦ использует уже спроектированный ABS Rates API — без дублирования логики.
- SFTP-выгрузка автоматизирует процесс и снимает ручную нагрузку с бэк-офиса.
- Кэш в КЦ защищает АБС от частых запросов при пиковых звонках (P4, P5, +R5).
- Решение масштабируется: при появлении API у партнёра можно добавить REST без смены источника.

### Отрицательные

- **Eventual consistency**: партнёр может отставать до 24 ч (расписание импорта) или до следующего cron банка.
- Дополнительный компонент **Rates Export Service** и эксплуатация SFTP.
- Необходимо согласование формата файла и тестирование с внешним контрагентом — риск задержки.
- При сбое SFTP партнёр работает на последнем файле; нужен операционный регламент (DT-3).

### Нейтральные

- Персонализированные ставки в КЦ остаются на этап 6 месяцев (F4).
- Внутренний КЦ по заявкам с сайта продолжает работать через существующую интеграцию с АБС (ADR-001).
