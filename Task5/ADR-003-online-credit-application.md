# ADR-003: Концептуальная архитектура онлайн-заявки на кредит

## Статус

Принято

## Дата

2026-06-06

## Контекст

После MVP по депозитам (ADR-001) банк «Стандарт» в течение **12 месяцев после MVP** запускает **онлайн-заявку на кредит** с сайта и из интернет-банка. Цель — предварительное решение **за один рабочий день** (F16, R6), продолжение той же заявки в отделении (F18).

**Текущее состояние (AS-IS):**
- Заявку на кредит принимают только в отделении через АБС.
- АБС ↔ Кредитный конвейер (Camunda BPM, Oracle, Java) — **пакетный обмен 1 раз/сут** (+R9); полный цикл — около **недели**.
- Скоринг (Python, Flask, PostgreSQL) и БКИ доступны по REST, но **сайт и ИБ не участвуют** в процессе.
- Интернет-банк для платежей обращается к БД АБС напрямую (R3); для новых процессов это недопустимо.
- Интернет-банк — монолит ASP.NET MVC, ядро у подрядчика (+R6); новый функционал выносится за пределы ядра (S2).

**Силы, влияющие на решение:**
- SLA «решение за день» несовместим с суточным batch +R9 — нужен **онлайн-маршрут** в конвейер (+R10).
- На сайте о клиенте известны **только данные БКИ** (+R14); в ИБ — полный профиль в АБС.
- Скоринг нельзя нагружать предрасчётами при каждом просмотре (R5); предодобрения (F19) — **ночной batch**.
- Полные условия продуктов — в **Кредитном конвейере** (F13), не в АБС.
- Заявки из ИБ передаются в конвейер **асинхронно** через Kafka (R4); синхронно — только скоринг и БКИ.
- Переиспользуются Kafka и SMS Confirmation Service из ADR-001 (S3).

## Ограничения решения

- **Скоринг с сайта** ограничен данными БКИ (+R14); полный профиль клиента доступен только в ИБ.
- **Суточный batch +R9** между АБС и конвейером не устраняется для офлайн-заявок; digital-канал обходит его через Conveyor Online Adapter.
- **Ядро ИБ** у подрядчика (+R6): UI кредитов встраивается в монолит, бизнес-логика — в микросервисах банка.
- **Выдача кредита** без визита в отделение — вне scope; идентификация клиента остаётся офлайн.
- **Eventual consistency** статуса заявки между каналами, конвейером и АБС до синхронизации batch.

## Рассмотренные варианты

1. **Только batch-маршрут АБС → Конвейер (+R9)** — как сейчас, но с заявкой с сайта/ИБ.
2. **Прямая интеграция каналов с БД АБС** — синхронные запросы к Oracle.
3. **Синхронный REST только через API Кредитного конвейера** — все вызовы через Camunda.
4. **Credit Application Service + REST к скорингу/БКИ + Kafka к онлайн-адаптеру Конвейера** — микросервис банка вне монолита ИБ.

## Решение

Мы реализуем **вариант 4**: **Credit Application Service** [.NET, микросервис банка] оркестрирует заявки с сайта и ИБ; скоринг и БКИ — **REST напрямую** (+R10); передача заявки в конвейер — **асинхронно через Kafka** (R4) и **Conveyor Online Adapter**, минуя суточный batch для digital-канала.

### Участвующие IT-команды

| Команда | Ответственность |
|---------|----------------|
| **Команда цифровой трансформации** | Координация, каталог и форма заявки на сайте |
| **Команда интернет-банка (банк)** | Credit Application Service, Credit Notification Service, кэши, интеграция с монолитом |
| **Команда интернет-банка (подрядчик)** | Точки расширения монолита (UI кредитов), без изменений ядра |
| **Команда Кредитного конвейера** | Camunda workflow, Conveyor Online Adapter |
| **Команда кредитного скоринга** | REST API скоринга, интеграция с БКИ |
| **Команда АБС** | Единый ID заявки, работа отделения (F18) |
| **Команда сайта / маркетинга** | UI каталога и формы (PHP, React.js) |

### Диаграмма контекста (C4, Level 1)

Файл: [`c4-context-credit-online.drawio`](c4-context-credit-online.drawio)

Система **«Онлайн-заявка на кредит»** — приём заявок с сайта и из ИБ, скоринг, передача в Кредитный конвейер, уведомления клиента.

**Акторы:**
- **Потенциальный клиент** — просматривает предложения и подаёт заявку на сайте (UC-16, UC-17).
- **Клиент банка** — видит предодобрения и подаёт быструю заявку в ИБ (UC-21, UC-22, UC-24).
- **Менеджер отделения** — продолжает заявку по единому ID (UC-26).

**Внешние системы:**
- **Сайт банка** — канал привлечения (PHP, React.js).
- **Интернет-банк** — канал самообслуживания (ASP.NET MVC, MS SQL).
- **Кредитный конвейер** — workflow и полные условия продуктов (Camunda BPM, Java).
- **Кредитный скоринг** — предварительный и повторный скоринг (Python, Flask).
- **АБС** — учётная система, работа отделения (Delphi, Oracle).
- **БКИ** — внешний источник данных о клиенте (+R14).
- **СМС-шлюз** — доставка СМС клиентам.

**Ключевые связи:**
- Клиенты → Сайт / ИБ: HTTPS (TLS, +R7).
- Сайт → Конвейер: каталог предложений (REST, кэш).
- Сайт / ИБ → Credit Application Service: заявка на кредит.
- Credit Application Service → Скоринг → БКИ: REST (+R10).
- Credit Application Service → Kafka → Конвейер: заявка (async, R4).
- Конвейер → АБС: batch 1/сут для legacy (+R9).
- Credit Application Service → СМС-шлюз: уведомления (UC-20, UC-25).

### Диаграмма контейнеров (C4, Level 2)

| Контейнер | Назначение | Интеграции |
|-----------|------------|------------|

| Система | Контейнер | Интеграции |
|---------|-----------|------------|
| **АБС** | Клиентское приложение АБС (Delphi) | Единый ID заявки, UC-26, F18 |
| **Kafka** | Брокер сообщений | ← Credit Application Service; → Conveyor Online Adapter |
| **БКИ** | Внешняя система | ← Scoring API |
| **СМС-шлюз** | Внешний провайдер | ← SMS Confirmation Service; ← Credit Notification Service |

### Основные потоки

**Поток A — заявка с сайта (UC-16 → UC-17 → UC-18 → UC-19 → UC-20):**
1. Клиент видит предложения (Offers Cache ← Конвейер).
2. Клиент отправляет заявку → Credit Application Service.
3. Сервис вызывает Scoring API → БКИ (+R14, +R10).
4. При одобрении — событие `CreditApplicationSubmitted` в Kafka → Conveyor Online Adapter → Camunda.
5. СМС клиенту: решение (UC-19) или визит в отделение (UC-20).

**Поток B — быстрая заявка из ИБ (UC-21 → UC-22 → UC-23 → UC-24 → UC-25):**
1. Клиент видит предодобрения (Pre-approval Cache, ночной batch, R5).
2. Клиент подаёт заявку → Credit Application Service (U3).
3. Повторный скоринг; при неизменности — упрощённый путь (F21).
4. SMS Confirmation Service проверяет код (UC-24).
5. Kafka → Conveyor Online Adapter; Credit Notification Service шлёт статусы (F23).

**Поток C — отделение (UC-26, F18):**
1. Конвейер синхронизирует статус с АБС (batch 1/сут, +R9).
2. Менеджер отделения продолжает заявку по **creditApplicationId** в АБС.

## Последствия

### Положительные

- SLA «решение за день» за счёт онлайн-маршрута (+R10) в обход суточного batch.
- Единый ID заявки от сайта/ИБ до отделения (F18).
- Каналы не обращаются к Oracle АБС напрямую (R3).
- Credit Application Service вне ядра подрядчика (S2, +R6).
- Скоринг защищён от лишней нагрузки: batch для предодобрений, REST только при заявке (R5).

### Отрицательные

- Два маршрута интеграции с Конвейером (online + legacy batch +R9).
- Eventual consistency статуса между каналами и АБС.
- Зависимость от доработки Conveyor Online Adapter.
- Скоринг с сайта только по БКИ — ниже точность (+R14).

### Нейтральные

- Batch АБС ↔ Конвейер для офлайн-заявок не меняется (+R9).
- Kafka и SMS Confirmation переиспользуются из ADR-001.

## Подтверждение

- Диаграммы C4 покрывают UC-16 … UC-26 и F13–F23.
- Согласование с командами конвейера, скоринга и интернет-банка.
- NFR: нет прямых запросов каналов к Oracle; TLS на внешних каналах (+R7).
