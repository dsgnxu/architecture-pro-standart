### **Решение**

Приведите диаграммы контекста и контейнеров в модели C4. Опишите там основные компоненты и интеграции всех элементов решения.

Также опишите, какой логикой вы руководствовались в ходе принятия решений и выбора технологий. Не забывайте, что необходимо учесть все функциональные и нефункциональные требования.

**Выбранный подход:** **Credit Application Service** (.NET, микросервис банка) оркестрирует заявки; скоринг и БКИ — **REST напрямую** (+R10); передача в конвейер — **асинхронно через Kafka** (R4) и **Conveyor Online Adapter**, обход суточного batch для digital-канала.

**Логика принятия решений:**
- SLA «решение за день» несовместим с batch +R9 — нужен онлайн-маршрут через Conveyor Online Adapter (+R10).
- Каналы не обращаются к Oracle (R3) — заявки через Kafka, скоринг через REST.
- Ядро ИБ у подрядчика (+R6) — бизнес-логика в микросервисах банка (S2).
- R5: Scoring API не получает нагрузку от предрасчётов — предодобрения рассчитываются offline (ночной batch) и отдаются из кэша; синхронный REST к скорингу только при подаче заявки.
- Kafka и SMS Confirmation Service переиспользуются из Task 3 (S3).
- Полные условия продуктов — в Кредитном конвейере (F13), не в АБС.

#### Диаграмма контекста (C4, Level 1)

Файл: [`c4-context-credit-online.drawio`](c4-context-credit-online.drawio)

**Акторы:** потенциальный клиент, клиент банка, менеджер отделения.

**Системы:** Сайт, Интернет-банк, Кредитный конвейер (Camunda BPM), Кредитный скоринг, АБС, БКИ [External], СМС-шлюз [External].

**Ключевые интеграции:**
- Сайт / ИБ → Credit Application Service: REST, заявка.
- Credit Application Service → Scoring API → БКИ: REST (+R10).
- Credit Application Service → Kafka → Conveyor Online Adapter → Camunda.
- Конвейер → АБС: batch 1/сут (+R9) для legacy-синхронизации.

#### Диаграмма контейнеров (C4, Level 2)

Файл: [`c4-container-credit-online.drawio`](c4-container-credit-online.drawio)

| Система | Контейнер | Назначение |
|---------|-----------|------------|
| **Сайт** | Веб-приложение сайта | Каталог кредитов, форма заявки |
| **ИБ** | Монолит ИБ (ядро подрядчика) | Auth, точка встраивания UI кредитов |

**Основные потоки:**
- **С сайта:** каталог → заявка → скоринг/БКИ → Kafka → конвейер → СМС → отделение.
- **Из ИБ:** предодобрения → быстрая заявка → СМС-код → скоринг → Kafka → конвейер.
- **Отделение:** batch-синхронизация конвейер ↔ АБС → менеджер продолжает заявку.

### **Альтернативы**

Опишите здесь наиболее важные альтернативные решения.

1. **Только batch-маршрут АБС → Конвейер (+R9)** — как AS-IS, но с заявкой с сайта/ИБ. Отклонено: SLA «решение за день» невыполним (неделя цикла).
2. **Прямая интеграция каналов с БД АБС** — синхронные запросы к Oracle. Отклонено: нарушает R3, перегрузка БД.
3. **Синхронный REST только через API Кредитного конвейера** — все вызовы через Camunda. Отклонено: избыточная связность, нагрузка на конвейер при каждом просмотре.
4. **Credit Application Service + REST к скорингу/БКИ + Kafka к адаптеру Конвейера** — **принято.** Баланс R3, R4, R5, R6, +R10.

### **Недостатки, ограничения, риски**

Подробно опишите здесь недостатки, ограничения и риски выбранного решения.

**Недостатки:**
- Два маршрута интеграции с Конвейером: online (Kafka) и legacy batch (+R9).
- Eventual consistency статуса между каналами, конвейером и АБС.
- Зависимость от доработки Conveyor Online Adapter командой конвейера.
- Скоринг с сайта только по БКИ — ниже точность решения (+R14).

**Ограничения:**
- Выдача кредита без визита в отделение — вне scope; идентификация остаётся офлайн.
- Batch АБС ↔ Конвейер для офлайн-заявок из отделения не меняется (+R9).
- UI кредитов встраивается в монолит подрядчика; бизнес-логика — только в микросервисах банка (+R6).
- Полные условия продуктов доступны только в Кредитном конвейере (F13).

**Риски:**
- Задержка согласования Conveyor Online Adapter с командой конвейера.
- Рост очереди Kafka при недоступности конвейера.
- Рассогласование статуса заявки между digital-каналами и АБС до nightly batch.
- Зависимость от доступности БКИ для скоринга новых клиентов с сайта.
